## 一、产品概述
### 1.1 产品定位
Cognix 是**面向企业办公场景的主动式个人工作习惯记忆引擎**，通过观察用户在飞书、CLI 等办公场景的行为信号，自动学习工作习惯、使用偏好与周期规律，在合适时机主动提供智能协助，区别于通用记忆层的被动存储检索，Cognix 核心价值是"主动理解用户工作模式，在正确时间提供正确帮助"。

### 1.2 目标用户
- 企业知识工作者（研发、产品、运营等）
- 飞书深度用户
- 高频使用 CLI 的技术人员

### 1.3 价值主张
| 用户痛点 | 解决方案 | 价值 |
|----------|----------|------|
| 偏好需要反复向 AI 说明 | 自动记忆长期办公偏好 | 减少重复沟通成本 |
| 重复性工作需要手动触发 | 挖掘周期性行为规律，主动提醒/执行 | 办公效率提升 80%+ |
| AI 无法理解个人工作节奏 | 动态学习习惯变化，自适应调整规则 | 个性化体验增强 |

## 二、竞品分析
### 2.1 调研产品对比
| 维度 | Mem0（https://github.com/mem0ai/mem0） | ReMe（https://github.com/agentscope-ai/ReMe） | Cognix |
|------|---------------------------------------|---------------------------------------------|--------|
| 定位 | 通用 AI Agent 记忆层 | 智能体记忆管理框架 | 企业办公场景主动记忆引擎 |
| 核心能力 | 记忆存储、混合检索（语义+BM25+实体匹配） | 上下文压缩、多模态记忆存储 | 行为规律挖掘、规则生成、主动触发服务 |
| 交互模式 | 被动调用（上层业务主动读写记忆） | 被动调用（Agent 推理前钩子调用） | 主动服务（无需用户触发，时机成熟主动推送） |
| 场景适配 | 全场景通用（客服、助手、医疗等） | 通用 Agent 场景 | 垂直聚焦飞书/CLI 办公场景 |
| 部署方式 | 云服务/自托管/SDK 库 | 本地库/自托管 | 本地可执行文件/飞书 Bot |
| 记忆可见性 | 黑盒（数据库存储，不可直接编辑） | 白盒（Markdown 文件存储，可直接编辑迁移） | 白盒（规则/偏好可视化，用户可手动修改） |
| 优势 | 生态成熟、多平台 SDK、生产级可用 | 轻量化、无额外依赖、记忆可解释性强 | 垂直场景优化、主动服务、办公场景深度适配 |
| 劣势 | 通用场景无垂直优化、被动调用无主动能力 | 功能较基础、主动能力缺失 | 场景相对聚焦 |

### 2.2 可借鉴特性
#### 从 Mem0 借鉴：
1. **混合检索机制**：语义+BM25 关键词+实体匹配的多信号检索算法，提升偏好召回准确率
2. **CLI 工具设计**：`mem0 add`/`mem0 search` 等命令行交互模式，适合技术用户使用
3. **Benchmark 框架**：成熟的记忆召回率、抗干扰、矛盾更新测试方案，可直接复用
4. **多租户支持**：按用户 ID 隔离记忆数据，为后续多用户扩展预留能力

#### 从 ReMe 借鉴：
1. **轻量化存储方案**：ReMeLight 的文件系统存储模式，无数据库依赖，MVP 阶段可快速落地
2. **记忆可解释性**：记忆以 Markdown/JSON 格式明文存储，用户可直接查看、编辑、迁移
3. **自动衰减机制**：长期未激活的偏好自动降低权重，适配用户习惯变化
4. **上下文压缩**：长对话/长行为日志自动摘要，减少 Token 消耗

### 3.2 核心用户场景
#### 场景 1：周报助手（P0）
- **前置条件**：用户连续 4 周周五 16:00 左右请求生成周报，默认格式为表格，接收人为直属 Leader
- **触发时机**：第 5 周周五 15:55
- **预期结果**：系统主动通过飞书/CLI 推送提醒：「今天是周五，已按你偏好的表格格式生成本周周报草稿，是否发送给 Leader A？」
- **后续流程**：用户点击确认直接发送，点击修改则调整内容/格式/接收人，系统自动学习新偏好。

#### 场景 2：会议助手（P1）
- **前置条件**：用户日历每周一 10:00 有团队例会，会前 5 分钟习惯查看上周待办
- **触发时机**：每周一 9:55
- **预期结果**：系统主动推送：「距离例会还有 5 分钟，已整理上周待办完成情况，需要打开文档吗？」

#### 场景 3：输出偏好自适应（P1）
- **前置条件**：用户连续 3 次选择表格格式输出内容
- **预期结果**：后续所有输出请求默认使用表格格式，无需用户重复说明，可手动切换。

#### 场景 4：CLI 命令推荐（P1）
- **前置条件**：用户高频使用 `report summary --weekly --table` 命令
- **预期结果**：用户输入 `report` 前缀时，自动补全推荐完整命令，一键执行。

## 四、非功能需求
1. **性能要求**：
   - 偏好查询响应时间 < 100ms
   - 规则触发延迟 < 1s
   - 支持单用户至少 1000 条行为数据存储
2. **安全要求**：
   - 所有行为数据本地存储，不上云（MVP 阶段）
   - 仅采集授权范围内的行为数据，支持一键清除所有数据
   - 敏感信息（如联系人、文档内容）加密存储
3. **兼容性要求**：
   - CLI 工具支持 Windows/macOS/Linux 三大平台
   - 飞书 Bot 适配飞书开放平台最新版本
   - 支持接入不同大模型（通义千问、GPT、Claude 等）
4. **可扩展性要求**：
   - 模块解耦，后续可扩展支持企业微信、钉钉等其他办公平台
   - 支持插件机制，可自定义规则触发后的动作

## 五、已确认决策点
| 序号 | 问题 | 最终选择 | 说明 |
|------|------|----------|------|
| 1 | MVP 部署形态选择 | B. Python 库，通过 pip 安装（同时向外暴露 CLI） | 兼顾安装便捷性与使用灵活性，支持作为库集成到其他 Agent 系统 |
| 2 | LLM 依赖策略 | 与 Mem0/ReMe 类似，作为记忆系统集成到 Agent，使用 Agent 系统的 LLM | 核心逻辑无强 LLM 依赖，仅在生成内容时调用上层提供的 LLM 能力，降低耦合 |
| 3 | 行为数据采集边界 | ABC 全部采集 | 同时支持显式上报、Shell Hook 自动采集 CLI 命令、飞书日程/文档接入，用户可按需开关 |
| 4 | 飞书接入优先级 | B. MVP 阶段同时开发 CLI 和飞书 Bot | 保证演示效果完整性，覆盖技术用户与普通办公用户两类人群 |
| 5 | 规则生效策略 | B. 新规则生成后先推送用户确认，同意后生效 | 降低误触风险，保证用户控制权，符合主动服务不打扰的原则 |
| 6 | 多用户支持 | A. MVP 阶段仅支持单用户使用 | 降低架构复杂度，聚焦核心场景验证，后续版本再扩展多租户能力 |
