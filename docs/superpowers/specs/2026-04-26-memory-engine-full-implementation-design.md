# 记忆引擎完整实现设计文档

## 概述
本设计文档描述了如何在现有Cognix记忆系统基础上，扩展实现面向飞书办公场景的外挂式记忆引擎完整功能，采用渐进式迭代方案，保持100%向下兼容，分6个阶段逐步实现。

## 目标
构建一套五层架构的记忆引擎，为飞书OpenClaw、飞书Bot、CLI Agent等AI助手提供长期记忆、短期上下文、工作习惯理解、流程复用与主动提醒能力，实现"越用越懂用户、越用越会做事"的持续进化。

## 架构设计
### 五层架构
```
┌─────────────────────────────────────────────────────────┐
│                服务执行层（新增）                        │
│  OpenClaw Hooks / 飞书 Bot / CLI 工具                    │
├─────────────────────────────────────────────────────────┤
│                智能决策层（新增）                        │
│  记忆压缩流水线 / 工作流沉淀 / 偏好理解 / 主动提醒         │
├─────────────────────────────────────────────────────────┤
│                长期记忆层（改造现有）                    │
│  从单MEMORY.md → 拆分为分类存储：                        │
│  user.md / settings.md / office.md / feedback.md /      │
│  reference.md / sessions/ / skills/                     │
├─────────────────────────────────────────────────────────┤
│                短期记忆层（改造现有）                    │
│  从内存存储 → Redis存储，支持TTL、Token感知压缩          │
├─────────────────────────────────────────────────────────┤
│                数据接入层（新增）                        │
│  标准事件结构 / 多源数据接入 / 权限隔离                   │
└─────────────────────────────────────────────────────────┘
```

### 兼容策略
- 所有现有API保持100%兼容
- 新老数据格式共存，自动迁移
- 新增功能通过新接口提供，按需启用

## 分阶段实现方案

### 阶段1：长期记忆分类改造
**目标**：实现长期记忆分类存储，替代现有单MEMORY.md模式
**核心工作**：
1. 定义5类长期记忆文件结构：
   - user.md：用户画像（常用语言、输出风格、工作角色、高频协作者、常关注项目）
   - settings.md：习惯性设置（视图偏好、通知设置、文档模板、输出格式）
   - office.md：办公习惯（工作流程、周期性任务、联系人生态）
   - feedback.md：纠偏反馈（用户对AI输出的长期修正意见）
   - reference.md：长期资源（项目地址、文档链接、团队规范）
2. 新增两个目录：
   - sessions/：历史会话存档
   - skills/：沉淀的可复用工作流
3. 兼容处理：
   - 保留现有MEMORY.md读取支持，自动迁移内容到对应分类文件
   - 新增接口：`add_classified_memory(category, heading, content)`
   - 现有`add_persistent_memory()`接口自动映射到合适的分类

**验收标准**：
- 新分类文件自动创建
- 现有MEMORY.md内容正确迁移
- 所有现有接口功能不受影响
- 新分类写入功能正常

### 阶段2：数据接入层实现
**目标**：实现统一的多源数据接入入口
**核心工作**：
1. 定义标准事件结构：
```python
{
    "user_id": str,
    "time": datetime,
    "source": str,  # openclaw / agent / feishu / user_input
    "event_type": str,  # conversation / tool_call / calendar / todo / etc.
    "content": str,
    "metadata": dict
}
```
2. 实现事件总线：
   - 支持事件订阅与分发
   - 插件式数据源接入
   - 权限隔离机制
   - 增量同步能力
3. 首批接入数据源：
   - OpenClaw对话历史
   - Agent执行轨迹
   - 用户主动输入

**验收标准**：
- 标准事件结构定义完成
- 事件总线正常工作
- 首批数据源接入成功
- 事件持久化与分发正常

### 阶段3：短期记忆Redis改造
**目标**：将短期记忆从内存迁移到Redis，提升性能与可靠性
**核心工作**：
1. 实现Redis短期记忆存储：
   - 会话级数据隔离
   - TTL自动过期机制
   - Token感知压缩算法
2. 短期记忆内容：
   - 当前会话最近若干轮消息
   - 当前任务目标与阶段状态
   - 最近工具执行结果
   - 当前临时偏好
   - 上下文资源（附件、文档、链接）
3. 会话结束处理：
   - 自动触发记忆筛选流程
   - 高价值内容写入长期记忆

**验收标准**：
- Redis连接与存储正常
- TTL过期机制正常
- Token压缩功能正常
- 会话结束自动筛选写入正常

### 阶段4：记忆压缩流水线实现
**目标**：实现高价值记忆自动提取与写入，避免记忆污染
**核心工作**：
1. 触发条件：
   - 会话轮数达到阈值（默认20轮）
   - Token接近上下文上限（默认80%）
   - 任务完成
   - 用户主动要求保存
2. 处理流程：
   - 会话内容 → 会话摘要 → LLM提取长期价值信息 → 分类校验 → 写入对应Markdown文件
3. 写入约束：
   - 仅允许写入5类长期记忆文件
   - 禁止生成无关内容
   - 支持用户审核确认

**验收标准**：
- 自动触发机制正常
- LLM提取与分类准确
- 写入格式符合规范
- 无冗余或无关内容写入

### 阶段5：双轨检索机制实现
**目标**：实现自动记忆检索与上下文增强
**核心工作**：
1. 固定核心记忆注入：
   - 每次任务默认注入user.md摘要、settings.md、office.md高频规则
2. 动态相关记忆召回：
   - 使用SQLite FTS5对feedback.md、reference.md、sessions/、skills/进行全文检索
   - 召回Top5最相关内容加入上下文
3. 检索优化：
   - 结果可解释、可控
   - 检索速度 < 100ms
   - 无需向量数据库

**验收标准**：
- 固定记忆注入正常
- 动态召回准确率达标
- 检索性能符合要求
- 上下文增强效果明显

### 阶段6：上层能力实现
**目标**：实现完整的上层功能与集成
**核心工作**：
1. 工作流沉淀模块（Skills Manager）：
   - 实时识别可复用流程
   - 后台归纳重复行为
   - 生成工作流草稿供用户确认
2. 主动提醒与飞书交互模块：
   - 定时任务调度
   - 飞书卡片交互
   - "主动提醒 + 用户确认执行"模式
3. 服务执行层集成：
   - OpenClaw Hooks集成（before_prompt/after_response/task_finish）
   - 飞书Bot实现
   - CLI工具实现

**验收标准**：
- 工作流识别与沉淀正常
- 主动提醒与飞书交互正常
- 所有集成方式工作正常
- 完整端到端流程测试通过
