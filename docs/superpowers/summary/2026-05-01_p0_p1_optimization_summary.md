### 2. 遇到的问题与破局方法
| 问题 | 破局方法 |
|------|----------|
| SQLite表创建报错`unrecognized token: "#"` | SQLite不支持`#`注释，全部替换为`--`格式的SQL标准注释 |
| 多跳递归查询循环引用报错`circular reference` | 增加`visited`字段记录已经访问过的实体ID，避免循环遍历，同时加异常捕获，查询失败返回空不影响主流程 |
| 测试脚本方法不存在`note_short_term` | 直接操作`short_term_memory`字典添加测试数据，避免依赖上层接口 |
| 懒加载测试除零错误 | 增加边界判断，首次读取时间为0时跳过倍数计算，直接判断缓存是否生效 |
| 多跳查询返回空 | 先检查实体是否存在，不存在直接返回空，避免无效查询 |

## 四、本周随手记 ✍️
1. **踩坑记录**：
   - SQLite注释只能用`--`，不要用Python风格的`#`，否则会语法错误
   - 递归查询一定要加防循环逻辑，否则很容易出现死循环
   - 新增功能一定要做开关，默认关闭，避免影响现有用户，灰度上线更安全
2. **心得**：
   - 纯Markdown存储的优势真的很大，用户可控性、可迁移性、可理解性远超所有基于数据库的记忆系统，这是我们最核心的差异化竞争力
   - SQLite的能力比想象中强很多，递归查询、全文检索、KV存储都能搞定，90%的场景不需要额外引入其他数据库
   - memU的设计有很多值得借鉴的地方，但我们不需要完全照搬，结合我们的核心优势做出来的产品比它更好用
3. **好用的写法**：
   ```python
   # SQLite递归多跳查询模板
   WITH RECURSIVE entity_hierarchy AS (
       SELECT id, name, 0 as depth, CAST(',' || id || ',' AS TEXT) as visited
       FROM entities WHERE id = ?
       UNION ALL
       SELECT e.id, e.name, eh.depth + 1 as depth, eh.visited || e.id || ',' as visited
       FROM entities e
       JOIN entity_relations r ON 
           (r.from_entity_id = e.id AND r.to_entity_id IN (SELECT id FROM entity_hierarchy))
           OR (r.to_entity_id = e.id AND r.from_entity_id IN (SELECT id FROM entity_hierarchy))
       WHERE eh.depth < ? AND eh.visited NOT LIKE '%,' || e.id || ',%'
   )
   SELECT DISTINCT * FROM entity_hierarchy WHERE depth > 0
   ```
